内存支持战之攻击模型

Posted on 五月 13, 2017

作者：Diting0x

CSysSec注： 本系列文章主要介绍内存的安全性问题，以及相应的攻击模型及防御措施，对整个系统安全问题的概览有很大的帮助。
转载本文请务必注明，文章出处：《内存持久战之攻击模型》与作者信息：Diting0x

  • 0X01 基本攻击模型
  • 0X02 构建攻击模型

内存破坏是计算机安全中亘古不变的话题。内存破坏也被看作是top three危险的软件错误。像C/C++这种底层语言很容易引来这些bugs. 这些程序语言无法对内存安全性的保证，以至于让一些攻击者利用内存bugs改变程序行为或甚至控制整个控制流（control-flow). 要了解什么是内存安全性，可参考上一篇文章：内存持久战-内存安全性. 然而，一种最显而易见的解决方案就是避免这些容易引起内存破坏的程序语言，用一种内存安全的程序语言去重写这些程序。显然这是不切实际的，比如重写整个OS？
尽管papers的许多防御措施层出不穷，但大部分都无法应用到工业上，归结于以下问题：性能开销太大、与现有的程序特性并不兼容、不够健壮、不够完整、依赖于编译器或源码等等。
对内存的破坏与保护是一场持久战。本文重在总结利用内存破坏的相关攻击及其实施过程。

0X01 基本攻击类型

  • 代码破坏攻击(Code corruption): 攻击者在内存中覆盖程序代码，执行自己写入的代码.
  • 控制流劫持（Control-flow hijack）: 攻击者重写程序的返回地址或跳转地址，转而执行自己的代码（shell code).
  • 数据攻击（Data-only attack）: 攻击者并不会修改程序控制流，而是操作程序中的关键数据获取额外权限或泄露重要信息。
  • 信息泄露（Information leak attack）:任何类型的内存错误都有可能导致内存内容的泄露

注意，也许你会问，怎么没有缓冲区溢出攻击，format string攻击等？ 这些都只是实施细节，属于过程，最终目的都可划分为以上的四种攻击模型。可具体看下文的构建攻击模型。

0X02 构建攻击模型

攻击者要实施一次完整的攻击，首先要触发内存错误，具体可按照以下六个步骤进行：
注：Step 1&2 是上述攻击模型通用的过程，必不可少，目的是为了触发内存错误。至于 Step 3-6，有些攻击需要完整的步骤才能执行，有些则在前几个阶段就可以完成。

Step 1: 让指针无效。

要让指针无效，可以强制让指针越界（out of bounds)，引用越界指针引发空间错误（spatial error)。包括：触发未经过检测的空间分配失败错误（allocation failure),这时指针将变成空指针；在循环中不作边界检测，不断递增或递减数组指针，让指针指向数组边界之外的内存空间导致缓冲区溢出；索引越界，一般由整数溢出、截断、对齐、错误的指针映射导致。

或可让指针指向已经被删除的对象,这时指针称为悬挂指针(dangling pointer).比如，利用不正确的异常处理器（exception handler)回收对象，但并不重新初始化指向这个对象的指针。
引用悬挂指针引发时域错误（temporal error). 时域错误也称为use-after-free漏洞，因为引用（used)悬挂指针是在指针指向的内存区域已经被回收（freed)到内存管理系统之后. 大部分此类的错误指针在heap上分配的对象，但局部变量的指针分配给全局变量后会造成指针逃逸(escape)出局部作用域，在函数返回或栈上的局部变量被删除后逃逸的指针将会变成悬挂指针。

Step 2: 强制让程序引用（读或写）无效指针。

Step 3: 利用无效指针修改对象。修改的对象包括数据指针、代码指针、代码、数据变量，以及输出数据变量

Step 4: 偏离源程序执行。可将修改的对象转向攻击者特定的代码（code curruption attack)、也可指向shellcode或者gadget(指一些指令序列）的地址、修改变量特定值、解析输出的数据变量（information leak attack). 注：此时，code currupttion attack与information lead attack过程都已全部完成。

Step 5: 使用修改后的对象。包括利用间接跳转指令 call/jump 引用指针，利用返回指令引用指针，以及引用破坏后的数据变量（data-only attack).至此，数据攻击过程已全部完成。

Step 6: 执行恶意代码。包括重用现有的函数/gadgets以及执行注入的shellcode(control-flow hijack).至此，control-flow hijack过程已全部完成。

内存持久战之内存安全性

Posted on 五月 13, 2017

作者：Diting0x

CSysSec注： 本系列文章主要介绍内存的安全性问题，以及相应的攻击模型及防御措施，对整个系统安全问题的概览有很大的帮助。
转载本文请务必注明，文章出处：《内存持久战之内存安全性》与作者信息：Diting0x

  • IEEE Security&Privacy’13
  • 不访问未定义的内存
  • 无限间距
  • Pointers as capabilities

C语言中的buffer overflows, format string attacks等其它的一些vulnerabilities都有一个共同的问题：违背内存安全(Memory Safety)。本文主要讲述如何准确定义内存安全，为什么这些vulnerabilities违背了内存安全。 也为后续两篇文章内存持久战-攻击模型，内存持久战-防御措施做好铺垫。

IEEE Security&Privacy’13

发表在IEEE Security&Privacy’13的一篇SoK(Systematization of Knowledge)文章，Eternal War in Memory ，阐述了一种定义内存安全的通用方法。
Definition 1: 文中提到，一个程序的执行，只要不会出现以下内存访问错误，就是内存安全的：

  • 1.缓冲区溢出
  • 2.引用空指针
  • 3.释放后使用（use after free）
  • 4.使用未初始化内存
  • 5.非法释放已经释放过的指针或未分配的指针

维基百科 wikipedia page on memory safety 也有类似的定义。从定义来看，排除这些错误是内存安全本身的定义所导向的，而并非内存安全性的本质。那么，如何将这些错误统一起来？

不访问未定义的内存

只有当程序访问未定义的内存时才会产生内存错误，这块内存是在程序中没有具体分配的，例如，heap 的一部分（通过malloc)，stack(作为局部变量或者函数参数），又或者是静态数据区域（作为全局变量). George Necula 在他的CCured项目中（旨在为C程序实施内存安全性）提到，一个内存安全的程序从来不会去访问未定义的内存。我们可以假设，内存可以无限的大，大到内存地址从不会复用(reused).如此一来，被释放的内存（可以调用free 或者从函数返回的时候pop）从不会被重新分配，并且会永久的保持未定义状态。

Definition 2: 不访问未定义的内存就是内存安全的。
这种定义明显排除了error 2 和 error 3. 如果将allocated 的定义包括initialized，又可以排除error 4. 如果假设free只能在定义过的内存指针中调用，那又可以排除error 5.

不幸的是，Definition 2 并未排除缓冲区溢出错误，也就是error 1。 来看一个例子，假定一个标准stack 布局， 在此定义下，program 1 的执行会被认为是内存安全的：

Definition 2 允许 Program 1 通过是因为此程序是在合法分配的内存中写数据，甚至写的数据类型也是正确的。但实际上问题在于，数组buf 的溢出将数据写进了变量x 中，显然这是内存不安全的。

无限间距

将 Definition 2 延伸， Program 1 被看作是内存不安全的。只要加上这个假设： 内存区域分配的间距是无限大的。

Buf 和 x 的分配间距无限的大，buf[5] 将会访问 buf 区域的边界外部。边界外部是个未定义的内存区域，按照上述定义，就会产生错误。heap ,静态数据区域对溢出的处理方式类似。

尽管 Definition 2 是个很接近让人满意的定义，但事实并未如此。来看 Program 1 的变形 Program 2，也是一种缓冲区溢出, Definition 2 仍然会允许 Program 2 执行。

这里，缓冲区溢出发生在 object 的内部。我们仍然可以类似的在域间引入无限间距的概念来排除缓冲区溢出的错误。这并未太背离现实，C标准允许编译器决定不同域的间距。另一方面，程序语言把结构体当做一个单独的object (从 malloc 返回的单独指针). 许多程序会把一个结构体映射到另一个结构体，或者会确定好一种间距方案。许多编译器都支持这些操作，但是否可以有一种更好的定义不依赖于这些？

Pointers as capabilities

从 Definition 2 中，了解到许多概念，比如，定义的（分配的），未定义的（从没有分配的或者分配后回收的),我们假设分配后回收的内存不会再复用。如此一来，只要访问未定义的内存，就会违背内存安全性。

Definition 3: 我们引入这么一个概念, Pointers as capabilities. 也就是说，允许指针的持有者访问一定区域中的内存。一个指针由三个元素组成（p,b,e): b 定义有效的区域，e 定义边界，p 代表指针本身。 程序只能操作p，b 和e ,这样做只是为了定义一次执行是否是内存安全的。

在某种意义上来说，基于capability定义的内存安全性是一种类型安全形式（type safety)。这里只有两种类型：指针类型和非指针类型。这种定义保证了 1) 指针只在定义了合法内存区域的安全模式下被创建. 2) 指针只有在它们是指向分配给它们的内存区域的情况下被解引用. 3) 那块内存区域仍然是定义过的。这种定义排除了上述所有的五种错误。

系统安全浅薄知识系列(一)-ASLR

Posted on 五月 8, 2017

作者：@Diting0x

ASLR全称是Address Space Layout Randomization，翻译成中文就是地址空间布局随机化。

怎么做？

随机化程序的关键数据在内存中的位置，这些数据包括程序的数据段、堆、栈以及共享库。见下图：（图来源于CCS’15）

这些数据为什么没有包括代码段？ 下文会继续讨论。

干嘛用的？

ASLR是一种保护缓冲区溢出攻击的防御技术，通过增加内存攻击(如return-into-libc或ROP)预测目标地址的难度阻碍其实施攻击。见下图：(图来源于CODASPY’16)

攻击者欲跳转到目标地址0xbfffd5d8的shellcode中，但在执行时其地址被随机化到了0xbfffe3f8中，此时跳转到0xbfffd5d8会导致内存奔溃。

ASLR历史

请看下图：（图来源于Abusing Performance Optimization Weaknesses to Bypass ASLR， 2014 BlackHat USA）

如何开启或关闭ASLR?

以Linux系统为例，通过配置/proc/sys/kernel/randomize_va_space中的值来开启或关闭ASLR.
运行cat /proc/sys/kernel/randomize_va_space命令，若输出为0，表示没有随机化，所有都是静态的；若输出为1，表示保守随机化，共享库、栈、mmap()，VDSO以及堆都被随机化；若输出为2，表示全随机化，在1的基础上，增加了通过brk()方式内存管理的随机化。

关闭ASLR,可以运行echo 0 /proc/sys/kernel/randomize_va_space

Windows和Linux在ASLR机制上有所不同，想了解详情可以参考这里

编译成PIE?

说到这里，还遗留了一个问题，那就是随机化的数据为什么没有包括代码段？ 要想让ASLR机制有效，就必须保证程序在内存空间的所有区域都是随机化的。有任何一部分没有被随机化，都有可能被攻击者利用来定位有效的gadgets来实施攻击。实际上，ASLR对代码段(.text)是不作随机化的，只有当可执行文件在编译成PIE(Position Independent Executable，位置依赖可执行文件)时，代码段才会被随机化。

对于non-PIE(也就是没有添加编译选项PIE),尽管ASLR的级别设置为2，攻击者仍然能利用程序的.text段以及GOT/PLT(全局偏移表/过程连接表)实施攻击(比如return-to-got/plt、ROP)。

既然non-PIE的可执行文件不能享受ASLR机制带来的安全性，那ASLR又有何意义? 是不是可以把所有的程序在编译时都加上PIE选项？ 对Linux系统来说，是有选择性的编译PIE，所有的库文件都是默认编译成PIE的，而大部分程序默认是不会被编译成PIE的。详情可以参考这篇论文的发现Too much PIE is bad for performance。

说了这么多，PIE的原理是什么？文章提到的这么多专有名词，ROP, Return-to-libc,Return-to-GOT/PIT, gadgets都又是什么呢？有机会再谈。

控制流完整性-Mathias Payer

Posted on 五月 5, 2017

译者: Diting0x
来源: https://nebelwelt.net/blog/20160913-ControlFlowIntegrity.html#cfi
转载请注明： CSysSec:控制流完整性

从高层面上说，控制流完整性(Contro-Flow Integrity, 简称CFI) 将程序的控制流限制到有效的执行踪迹中。CFI在运行时监控程序的执行并将其状态与预先设定好的有效状态对比来执行这一性质。如果检测到一种无效状态，就会发出警告，通常是终止程序的执行。这样一来，CFI属于利用运行时监控器来检测特定攻击面(控制流劫持)的防御方法，之后在运行时标记出漏洞利用尝试。其它运行时监控器的例子包括：ASan（Address Sanitizer,针对空间内存奔溃)，UBSan（Undefined Behavior Sanitize，针对C/C++中的未定义行为, 以及DangNull(针对时间内存崩溃)。大部分的这类监控器针对开发环境，在测试程序时检测违背原则。另一方面，CFI是一种有效的防御机制，并且所有的现代编译器,比如GCC,LLVM以及微软Visual Studio都以一种低开销但不同安全性保证上实现了CFI机制。

CFI通过限制控制流转移的目标来检测控制流劫持。在一次攻击流劫持攻击中，攻击者将程序的控制流重定向到不会到达正常执行的位置，比如，在一种可替换环境下注入代码或者复用代码。

[1]的初始想法中的防御机制，也是第一个(闭源)原型,在2005年被提出。此后，提出并实现了许多CFI防御机制的代替品。尽管所有的这些替代品只是简单的改变了底层实现或者分析过程，它们都在尝试实现CFI策略。这篇文章的目标不是像我们在这篇文章中[2](阅读这篇文章可以看到CFI领域非常详细的相关工作)做的去说明每种CFI机制的差异性，也不是像我们在[3]中做的将CFI和这些替代品去做比较，而是解释CFI是什么，CFI能做什么以及CFI的局限性在哪。

任何的CFI机制都包括两个抽象模块:一个分析模块(通常是静态分析)用来(在不同精确度下)恢复程序的控制流图(CFG)和一个动态实现机制用来根据生成的CFG来限制控制流。

下面的样例代码展示了一个含有5个函数的简单程序。foo函数用一个函数指针，CFI机制注入一个前向边(forward-edge)和一个后向边(backward-edge)检测。函数指针要么指向bar要么指向baz.依赖于forward-edge分析，不同目标集合可以被允许通过。

控制流转移

一种架构上的指令可以分为控制流转移指令和计算指令。计算指令被顺序执行(一个紧接着一个）,然而控制流转移指令以一种特定的方式改变控制流，有条件地或无条件地将控制流重定向到一个特定的代码区域。

控制流转移指令可以进一步分为直接和间接控制流转移指令，间接控制流转移的目标依赖于运行时的值，比如某个寄存器或某个内存中的值，而直接控制流转移，其目标通常是指令本身中的最近偏移。

对直接控制流转移的保护是很直接的，在大部分架构上，让可执行代码是只读的，因此目标不会被攻击者修改(就算发生了任意内存崩溃，因为可写位必须被禁止)。这些直接控制流转移通过只读或者独立内存页面来保护，这种保护是通过底层硬件(内存管理单元)来实现(没有开销)的。CFI假设可执行代码是只读的，否则攻击者就能简单的覆盖代码并移除运行时监控器。

间接控制流转移可以进一步分为forward-edge控制流转移和backward-edge控制流转移。Forward-edge控制流将代码向前转移到一个新的位置并且在间接跳转中使用，间接调用指令被映射到源代码级别，比如switch语句，间接调用或者虚拟调用。Backward-edge被用来返回到之前forward-edge中使用的位置中，比如，当使用return指令从函数调用中返回时。为了简单点，我们不讨论中断、中断返回以及异常。

生成控制流图

一个CFG是涵盖所有程序有效执行的图。图中的结点是程序中控制流转移的位置，边代表可达到的目标。CFG是一种抽象概念，不同的CFI机制使用不同的方法(依赖可执行文件或源代码进行静态和动态分析)来生成CFG。

对于前向边，CFG列举所有可能的目标，通常是利用源代码的信息。C/C++中的switch语句就是一个很好的例子，不同目标是能通过统计知道的，并且编译器能生成一个固定跳转表并通过边界检测释放一个间接跳转来保证运行时使用的目标是switch语句中有效的目标。

对于通过函数指针的间接函数调用，由于目标不能提前知道的缘故，分析变得更复杂。常用的基于源代码分析的方法利用一个基于类型的方法并检查函数指针中使用的函数原型，列举所有匹配的函数。不同的CFI机制利用不同形式的类型相等性(比如任何有效的函数，相同参数数量的函数，相同签名(相同参数数量并且相同参数类型)的函数。在运行时，任意具有相同匹配签名的函数都是被允许的。

仅仅检查函数类型可能会违背一些冲突(比如函数是可达的但实际上从不会被调用)。这种分析可能会过大估计目标的有效集合。实际上，编译器能检查哪个函数是address taken（就是说有一行代码能生成函数的地址并且存储这个地址)
CFI机制通过将相同函数原型的集合与address taken函数的集合求交集可能会减少可允许目标的数量。

对于虚拟调用，比如C++中依赖于对象类型与类关系的间接调用，分析可以进一步利用对象的类型来限制有效函数，比如，所有对象构造器有相同的签名单只有相关类的子集构造器是可行的。

到目前为止，构造的CFG是无状态的，比如，没有考虑执行环境并且每个控制流转移与其它是独立的。一方面，在运行时只有一个目标是允许被任意可能的转移，也就是说目标地址当前存储在代码指针指向的内存区域中。另一方面，依赖于不同精确度的分析，CFG的构造在不同粒度中过度估计了有效目标的数量。有些机制将路径限制考虑在内并且利用动态分析的方法验证当前执行的有效性来检查(有限的深度)程序中执行的路径是否可行。由于动态追踪在运行时的开销问题，只有一小部分机制会检查这种路径环境。

实施CFI

CFI可以在不同级别上来实施。有时分析阶段(CFG构造)以及实施阶段会有交叉的部分(比如，当考虑路径限制时)。大部分机制有两个基本的机制，一个用来前向边转移，另一个用来后向边转移。

上图展示了CFI是如何通过执行运行时监控器(根据可允许目标构建的集合)来限定可能目标的集合。如果观察的目标不在集合内，程序将被终止。

对于前向边转移来说，代码通常利用一些相等性检查的形式注入。这种检查确保在运行时观察的目标在有效目标的集合内。这可以通过完整集合检查或简单的类型比较来实现(比如，哈希函数，然后检查当前目标的哈希值是否等于调用点中期待的哈希值)。函数的哈希可以嵌入到函数内部，函数之前或者一个不相交的metadata表中。

后向边转移更难保护，因为如果用同样的方法，攻击者可能会在函数返回时，将控制流重定向到任意有效的调用点。强的后向边保护会利用栈中之前调用的函数的执行环境。实施栈完整性的机制能确保任意后向边转移职能返回到最近之前的调用者。这种性质能通过将之前调用点存储在shadow栈中或者在栈中确保内存安全性来实现。比如，如果return指令不能被修改，那就能保证栈的完整性。后向边控制流的实现比前向边控制流的实现更“简单”，因为函数调用和返回形成了一种symbiotic关系，而这种关系能在防御的设计中被利用起来，比如，一个函数返回总是返回到之前函数调用的文职。这种关系在前向边中是没有的。

总结

如果能实施正确，CFI是个强防御机制，其能限定攻击者的自由。攻击者可能仍可以破坏内存，并且数据攻击仍然存在。对于前向边，一个强机制必须要考虑语言特定的语义来尽可能多的限定有效目标的集合。此外，对于前向边的大部分机制都是无状态的，这就允许攻击者将控制流重定向到任意一个CFG识别的有效位置。限制目标集合的大小能约束前向边中的攻击者。对于后向边，一种实施栈完整性的执行环境敏感的方法能保证完整的保护。

虚拟机自省技术-一个有新商业应用的安全性创造

Posted on 五月 5, 2017

译者：Diting0x
来源：https://blog.xenproject.org/2016/08/15/virtual-machine-introspection-a-security-innovation-with-new-commercial-applications/
转载请注明: CSysSec:虚拟机自省技术-一个有新商业应用的安全性创造

这篇由Xen工程主席Lars Kurth编写的文章最初发表在Linux.com.

几周前，Citrix和Bitdefender开启了XenServer 7 和Bitdefender Hypervisor Introspect技术研究，这项研究构成了Xen工程中Hypervisor的虚拟机自省研究（VMI）下部构造的第一个商业应用。在这篇文章中，我们会涉及一下两点：一是为何这项技术具有如此革命性，二是Xen工程区的成员和早期VMI的采用者（最受关注的LibVMI和DRAKVUF）是如何协调成功应用这项技术的。

在虚拟环境下发展安全挑战

今天，恶意软件在与反恶意软件相同的环境并利用与之相同的特权运行。这也是一个愈发凸显的问题。我在这篇Linux.com文章中与the Walking Dead（行尸走肉）的类比再次显现出帮助。让我们来看看传统反恶意软件是怎样与这个画面相契合，并且我们的类比是否应用于反恶意软件。

在Walking Dead，Wakers尸行者已控制地球，大肆蚕食剩余的人类。尸行者时时刻刻都活跃着，被声音吸引着，最终形成一个可能超出你防御范围的尸群。他们很是强壮，但却十分的木讷。正如我们在那个Linux.com的文章中探知到的，人们会犯错误，所以我们不可能一直避免尸行者侵犯我们的领地。

在这一类比中，我们将尸行者比作恶意软件。试想我们的虚拟主机是一个由个人住房（虚拟机）组成的村子，Hypervisor和网络提供将村子联系起来的基础设施（街道，篱笆，供电…）。

进入反恶意软件的世界：试想那些剩余的人类生存了一段时间并重新发展他们的技术，能够快速识别尸行者并修复带来的一切破坏。这可以替代巡逻队、闭路电视监控系统、警报墙/窗口等安全设施、用于抗击已发现尸行者的军队和修复所有毁坏的清理组。不幸的是，传统反恶意软件技术的现实只用于个人房屋内（亦称虚拟机），而且它们不在村里的街道。

更严重的是，直到最近恶意软件相对来说还比较愚钝。然而，这个情况在最近的几年有了显著的改观。我们的尸行者逐渐进化成更快更强更智能的 Wayward Pine的Abbies。换言之，恶意软件现如今已经能够入侵或破坏掉我们的安全机制。

我们所需要的是用于监视整个村落的卫星监视的替代品和在它们尝试并闯入我们的房屋时的远程破坏激光束。当然，我们还可以用这个新发现的性能来快速部署所需陆军和清理人员。这本质上是虚拟机内省技术给我们的承诺。它允许我们从操作系统外部解决安全问题，而不用依赖由地面呈现的不可靠的功能。下面会更详细的介绍这一问题。

从虚拟机自省技术到第一次商业应用：一个合作佳话

将虚拟机自省技术应用到Xen Hypervisor项目上的想法是在2007年在Georgis Tech以 [Tal Garfinkel与Mendel Rosenblum]在2003年的研究为基础的。Xen Hypervisor项目在2009年通过XenAccess和内存事件的API第一次将这项技术集成进来。在某种程度上，这是对VMware的VMsafe技术的一次回应，VMsafe技术在2008年被提出，由于这项技术在可扩展性上有很大的限制在2012年被废弃。Vmsafe被vShield替代，vShield是Vmsafe的有效子集，是一种基于agent，hypervisor-facilitated, 文件系统反病毒的解决方案。

在Xen项目内部，虚拟机自省技术能存活的原因在于此项技术强大的研究兴趣，并且一些专业的安全应用程序在安全性上牺牲一些性能是可以接受的。这最终导致LibVMI的诞生(2010)，这让这些API变得更容易访问。LibVMI提供了一种抽象能够最终将XEN的VMI功能的自己能暴露给其他的开源虚拟化技术，比如KVM和QEMU。

在2013年5月，Intel发型了Haswell CPU，其能够利用#VE和VMFUNC扩展在VMCS上维护512个EPT指针。这被证明是对VMI技术一次潜在的颠覆，可以让hypervisor控制并让硬件在VM之间实现强隔离，而且开销比之前还小，这也让来自Bitdefender, Cisco, Intel, Novetta, TU Munich和 Zentific的安全研究员和工程师们合作起来。从2014年到2015年，XenAccess和内存事件(mem-events)API被重构到了Xen Hypervisor项目的新的VMI子系统中，添加了atl2pm和其它硬件特性，并且支持ARM CPU， 在Xen 4.6版本中已经发行。

Citrix和Bitdefender合作将VMI技术带向市场： XenServer 7.0引入了基于Xen项目VMI接口构建的Direct Inspect APIs。用Bitdefender HVI实现，能安全地将自省能力赋予安全容器。

今天到底能做什么？

回到我们的类比：我们需要的是卫星监视系统的替代物来观察整个村庄。VMI能做到吗？在理论上：VMI将观察任意一个虚拟机的状态变为可能（村庄里的房子和它周围)，这些状态包括内存，CPU状态并且当虚拟机改变时候能接受事情。在实际上，如果不使用硬件特性的话，这样做的性能开行还是太大了。

在我们想象的世界里到处都是Walkers和Abbies，这就好比我们不能监控所有的东西，意味着我们要利用自己的资源集中关注一些高价值的东西。换句话说，我们需要集中关注系统界限(每个房子的最近区域)的可疑行为。

这种集中关注是通过监控可疑行为的敏感内存区域来实现的。当检测到恶意行为时，能有一种解决方案对进程状态(阻塞，杀死）或者VM状态(暂停，关闭)作出正确反映，与此同时，能在一个正运行的VM中直接收集并报告取证细节。

任何时候Abbie或者Walker要接近我们的房子，想象一下我们卫星上的激光束。在技术上来说，卫星和激光设施对应XenServer的Direct Inspect API，控制和监控我们数据的软件对应Bitdefenders Hypervisor Introspection。

很重要的一点是要强调利用hypervisor来提供硬件实现的隔离，监控和补救行为发生在外部。这意味着攻击者不能破坏卫星或者激光束。

当然，没有一种安全方案是完美的。如果可疑行为不会影响到VM的内存，监控软件可能不总是能检测到这种行为。这并不会削弱基于文件系统安全的地位。我们仍然需要保持警惕，没有一种完美的防御方法。在我们的村庄类比重，我们照样会被地下基础设施（比如隧道和运河）攻击。在本质上，这意味着我们必须将VMI技术和传统的反恶意软件同时利用起来。

VMI和传统的hypervisor-facilitated 反病毒方案(比如vShield)相比如何呢？在我们的类比重，这些方案需要对安装在房子里的所有监控设备(CCTV,报警们/报警窗)做集中管理，然而对事件的监控特别像集中在我们村庄大厅的一个安全控制中心。即使这种方法能极大地简化发生在虚拟机里面的监控和管理方式，并不会提供自省方法能提供的额外的保护。

结论

VMI技术的发展和它的首次开源以及商业应用都表明Xen项目社区是如何以创新的方式在发展，并能将革命性的方案带向市场。阅读代码的稀有性，从中学到一些东西，提问以及提供该井让安全研究者和像Citrix和Bitdefender这种的供应商将解决方案带向市场。

值得一提的是，硬件使能的安全技术发展的非常快：只有Intel的#VE和VMFUNC的扩展子集如今被应用到了VMI技术上。利用更多的硬件扩展,能让out-of-guest工具带来的安全性以及in-guest工具带来的性能同时看到希望。

更令人鼓舞的是，其它的供应商如 A1Logic, Star Lab和 Zentific在研究新的基于Xen项目的安全解决方案。另外，集中关注安全，基于Xen的OpenXT project 项目已经开始和Xen项目社区走的越来越近了，并保证未来在安全上的创新。
